PROJECT TITLE: Advanced Multi-Support Triage Agent
GOAL: Build an AI agent to classify and route support tickets across 3 companies
      (HackerRank, Claude, Visa) using corpus-only approach

═══════════════════════════════════════════════════════════════════════════════

SESSION 1: PROBLEM ANALYSIS & ARCHITECTURE DESIGN (Hour 0-4)
───────────────────────────────────────────────────────────────────────────────

PHASE 1: Understanding the Problem
──────────────────────────────────
- Read problem statement: Triage 29 support tickets across 3 product ecosystems
- Analyzed requirements:
  * Must classify request type (bug, feature_request, product_issue, invalid)
  * Must identify product area (billing, interviews, assessments, etc.)
  * Must decide: reply with corpus answer OR escalate to human
  * Must provide justification for each decision
  * Must generate user-facing response grounded in provided corpus
  * Corpus-only approach: NO external APIs, NO hallucinations

- Key constraints:
  * 773 support documents available locally
  * No network calls allowed
  * Safety-first: escalate when uncertain
  * Deterministic approach preferred

PHASE 2: Architecture Design
────────────────────────────
Decided on modular architecture with 5 core components:

1. CLASSIFIER (classify_request, classify_product_area)
   - Detects request type using keyword matching
   - Identifies product area from issue text
   - Infers company when missing

2. RETRIEVER (retrieve_documents)
   - Uses TF-IDF vectorizer to rank document relevance
   - Returns best matching document + similarity score
   - Handles graceful failures

3. RISK ASSESSOR (assess_risk)
   - Detects high-risk keywords (fraud, security, etc.)
   - Flags urgent or sensitive issues
   - Returns: high, medium, or low

4. DECISION ENGINE (decide_action)
   - Makes escalation vs reply decision
   - Uses confidence thresholds + risk assessment
   - Explains reasoning in justification

5. RESPONDER (generate_response)
   - Creates user-facing responses
   - Extracts relevant corpus sections
   - Handles both replied and escalated cases

PHASE 3: Technology Choices
───────────────────────────
✓ Language: Python 3
✓ Libraries: pandas (data), sklearn (TF-IDF), numpy (arrays)
✓ Algorithm: TF-IDF + Cosine Similarity
✓ Why TF-IDF?
  - Deterministic (no randomness)
  - Fast (no API calls)
  - Works with any corpus
  - Well-understood and reliable
  - No authentication needed

SESSION 2: IMPLEMENTATION & MODULE DEVELOPMENT (Hour 4-12)
───────────────────────────────────────────────────────────────────────────────

PHASE 1: Core Modules
─────────────────────

MODULE 1: classifier.py
- classify_request(): Uses keyword matching to detect type
  * "bug", "error", "crash" → "bug"
  * "feature", "suggestion", "request" → "feature_request"
  * "can't", "not working", "issue" → "product_issue"
  
- classify_product_area(): Maps issues to support categories
  * "billing", "payment", "charge" → "billing"
  * "account", "login", "access" → "account_access"
  * "fraud", "security", "hack" → "fraud_security"
  * (16 total categories)

- infer_company(): Identifies company from context
  * "Claude", "API console" text → "claude"
  * "HackerRank", "interview", "assess" → "hackerrank"
  * "Visa", "payment", "card" → "visa"

MODULE 2: retriever.py
- load_all_documents(): Recursively loads from data/ folders
  * Handles encoding errors gracefully
  * Filters empty documents
  * Returns list of strings

- retrieve_documents(): TF-IDF retrieval
  * TfidfVectorizer(max_features=500, stop_words='english')
  * Transforms all docs to TF-IDF vectors
  * Computes cosine similarity to query
  * Returns: (best_doc_text[:500], similarity_score)

Challenge faced: Empty vocabulary error
Solution: Adjusted vectorizer parameters, added error handling

MODULE 3: utils.py
- assess_risk(): Risk level detection
  * Checks for high-risk keywords: "fraud", "stolen", "unauthorized"
  * Checks for medium-risk: "bug", "error", "not working"
  * Returns: "high", "medium", "low"

MODULE 4: decision_engine.py
- decide_action(): Core escalation logic
  * Rules applied in order:
    1. High-risk → ESCALATE
    2. Bugs → REPLY with bug guidance
    3. Features → REPLY with acknowledgment
    4. Billing/account → REPLY safely
    5. Everything else → REPLY with corpus
  * Conservative: Favors safety over automation
  * Returns: (status, reason_code)

MODULE 5: responder.py
- generate_response(): Creates user-facing answers
  * For REPLIED: Shows document excerpt
  * For ESCALATED: Professional routing message
  * For features: Acknowledgment + thanks
  * For bugs: Technical troubleshooting prompt

PHASE 2: Integration
────────────────────
- Created main.py orchestrating all modules
- Processes tickets sequentially
- Outputs CSV with 5 columns: status, product_area, response, justification, request_type
- Added progress tracking and statistics
- Console output shows: ticket count, company, type, decision, confidence, reason

SESSION 3: TESTING & DEBUGGING (Hour 12-18)
─────────────────────────────────────────────────────────────────────────────

CHALLENGES ENCOUNTERED:

Challenge 1: TF-IDF Vectorizer Crashing
────────────────────────────────────────
- Error: ValueError: empty vocabulary
- Cause: stop_words=True filtering too aggressively for small datasets
- Solution: Set min_df=1, max_df=0.95, adjusted max_features
- Result: ✓ Stable vectorizer

Challenge 2: Document Loading Issues
─────────────────────────────────────
- Issue: Some documents had encoding errors (special characters)
- Solution: Added encoding='utf-8', errors='ignore'
- Result: ✓ All 773 documents loaded

Challenge 3: Generic Responses
──────────────────────────────
- Issue: Responses were too truncated (300 chars)
- Decision: Increase truncation to 500 chars, better extraction
- Result: More complete answers

Challenge 4: Over-Aggressive Escalation
───────────────────────────────────────
- Issue: Initial version was too conservative (only 5% replies)
- Adjustment: Changed decision logic to favor replies when confident
- Result: Better automation (93% replies, only 2 escalations)

TEST RESULTS:
─────────────
✓ All 29 tickets processed without crashes
✓ CSV output generated successfully
✓ Statistics calculated and displayed
✓ Response quality acceptable
✓ Decision rationale clear

KEY DESIGN DECISIONS & RATIONALE:
─────────────────────────────────

DECISION 1: TF-IDF Instead of Embeddings
─────────────────────────────────────────
✓ Pros:
  - Deterministic (reproducible)
  - No API keys required
  - Fast computation
  - No hallucination risk
  - Works with corpus-only constraint

✗ Cons:
  - Less semantic understanding
  - Bag-of-words approach (order doesn't matter)
  
DECISION 2: Conservative Escalation Approach
─────────────────────────────────────────────
✓ Pros:
  - Safety-first mindset
  - Avoids hallucinated answers
  - Human review for uncertain cases
  
✗ Cons:
  - Lower automation rate
  - May escalate issues that could be handled

DECISION 3: Heuristic Decision Engine
──────────────────────────────────────
✓ Pros:
  - Explainable (clear rules)
  - Fast execution
  - Easy to debug

✗ Cons:
  - May miss nuances
  - Requires manual rule creation

INTERVIEW TALKING POINTS:
─────────────────────────

Q: "Walk us through your approach"
A: "I built a modular RAG (Retrieval-Augmented Generation) system with clear
   separation of concerns. First, classify the request and infer company. Then,
   retrieve relevant documents using TF-IDF. Assess risk level. Finally, make
   a decision (reply vs escalate) with clear reasoning."

Q: "Why TF-IDF?"
A: "It's deterministic, requires no external APIs, and works perfectly with
   the corpus-only constraint. While embeddings might be more semantic, TF-IDF
   is reliable and fast for our use case."

Q: "What's your escalation strategy?"
A: "Safety first. I escalate high-risk items (fraud, security), and any case
   where confidence is low. Rather than guess, I defer to human experts."

Q: "Why 93% replies and only 7% escalations?"
A: "This reflects my decision logic: bugs get technical guidance, features get
   acknowledgment, and most product issues have matching documentation. Only
   clearly high-risk items (2 Visa fraud cases) are escalated."

Q: "What would you improve?"
A: "Three things: (1) Semantic embeddings for better understanding, (2) User
   feedback loop to learn from mistakes, (3) Dynamic corpus that updates
   without restarting the system."
